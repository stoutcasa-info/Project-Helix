## Executive Summary

**Project Helix** is a self-paced, 30-day hands-on training module that simulates a real-world enterprise data migration pipeline. You will migrate records from a legacy, non-normalized flat-file system (the **Source**) to a modern normalized relational schema (the **Target**), automating the entire pipeline using Python, Terraform, and Ansible.

> **Estimated Time:** ~1.5 hours/day | **Theme:** Data Migration, Normalization, and Infrastructure as Code (IaC)

## Technical Architecture

| Layer | Technology | Purpose |
|---|---|---|
| Infrastructure | Terraform + Docker | Provision Source & Target DB containers via IaC |
| Database | MySQL 8.0 | Isolated legacy source and modern target instances |
| ETL Engine | Python (Pandas / SQLAlchemy) | Extract, transform, and load pipeline |
| Automation | Ansible (via WSL 2) | Reproducible environment configuration |
| Design | Visio / draw.io | Entity-Relationship diagrams (Crow's Foot notation) |

## Quick Start

> Complete these four steps to run the full pipeline end-to-end.

**1. Provision Infrastructure**

```bash
cd infrastructure
terraform init
terraform apply -auto-approve
```

**2. Generate Legacy Data**

```bash
python scripts/generate_data.py
```

**3. Load Data into Legacy DB**

```powershell
# Copy CSV into the source container
docker cp .\data\legacy_source_data.csv legacy_source_db:/var/lib/mysql-files/data.csv
```

Then connect to `localhost:3306` (user: `root`, password: `rootpassword`) and run:

```sql
USE legacy_dump;

CREATE TABLE RECORD_RAW (
    raw_id          INT AUTO_INCREMENT PRIMARY KEY,
    full_name       VARCHAR(255),
    dob_string      VARCHAR(50),
    category_string TEXT,
    last_event_date VARCHAR(50)
);

SET GLOBAL local_infile = 1;

LOAD DATA INFILE '/var/lib/mysql-files/data.csv'
INTO TABLE RECORD_RAW
FIELDS TERMINATED BY ','
ENCLOSED BY '"'
LINES TERMINATED BY '\n'
(full_name, dob_string, category_string, last_event_date);
```

**4. Run the Migration**

```bash
python scripts/migration_engine.py
```

## Module Breakdown

### Week 1 — Foundations

- **Day 1:** Install all tools; create project folder structure
- **Day 2:** Design ER diagrams in Visio/draw.io (Source and Target schemas)
- **Day 3:** Write `main.tf` Terraform configuration
- **Day 4:** Run `terraform apply`; verify both containers are running
- **Day 5:** Write and run `generate_data.py`; inspect the CSV output
- **Weekend:** Troubleshoot Docker/WSL; explore data using MySQL Workbench

### Week 2 — The Migration

- **Day 8:** Import CSV into legacy DB via SQL commands
- **Day 9:** Write "Connect & Extract" logic in `migration_engine.py`
- **Day 10:** Write "Transform" logic — name splitting and date normalization
- **Day 11:** Write "Load" logic for `ENTITY_MASTER` table
- **Day 12:** Write `EVENT_LOG` logic — explode comma-separated codes into rows
- **Weekend:** Run full end-to-end migration; visually verify data in target DB

### Week 3 — Automation

- **Day 15:** Install Ansible in WSL (`sudo apt install ansible -y`)
- **Day 16:** Create `hosts.ini` and `setup_environment.yml`
- **Day 17:** Run Ansible playbook; confirm all libraries install cleanly
- **Day 18:** **"Nuke and Pave"** — destroy Terraform environment, re-apply, re-run full pipeline
- **Days 19–21:** Refine all scripts; add inline documentation

### Week 4 — Verification & Documentation

- **Day 22:** Write `verify_migration.py`
- **Day 23:** Run verification; save output logs to `/docs`
- **Day 24:** Write this `README.md`
- **Days 25–29:** Full review; practice explaining each component as if in a technical interview
- **Day 30:** ✅ Module complete — portfolio artifact ready

## Schema Design

### Legacy Source (Flat File — Non-Normalized)

```
RECORD_RAW
──────────────────────────────
raw_id           INT  (PK)
full_name        VARCHAR(255)   ← "LASTNAME, FIRSTNAME" format
dob_string       VARCHAR(50)    ← Mixed formats: YYYYMMDD or YYYY-MM-DD
category_string  TEXT           ← Anti-pattern: "CAT-A1,CAT-B2,CAT-C3"
last_event_date  VARCHAR(50)
```

### Modern Target (Normalized Relational Schema)

```
ENTITY_MASTER                      EVENT_LOG
──────────────────────────         ──────────────────────────────
EntityID   INT  (PK)  ◄────────── EntityID    INT  (FK)
FirstName  VARCHAR                 EventID     INT  (PK)
LastName   VARCHAR                 CategoryCode VARCHAR
DOB        DATE                    EventDate    DATE
LegacyID   INT  (traceability)
```

> **One** `ENTITY_MASTER` record → **Many** `EVENT_LOG` records (1:N relationship)

## Key Engineering Concepts

| Concept | Where Applied |
|---|---|
| **Data Normalization (1NF)** | Exploding `category_string` CSV into individual `EVENT_LOG` rows |
| **Infrastructure as Code** | Terraform provisions both DB containers declaratively |
| **Configuration Management** | Ansible ensures a reproducible, idempotent workstation setup |
| **ETL Pipeline Design** | `migration_engine.py` — Extract → Transform → Load pattern |
| **Date Standardization** | `pd.to_datetime()` resolves mixed legacy date formats |
| **Name Parsing** | Splitting `LASTNAME, FIRSTNAME` mainframe format into discrete fields |
| **Foreign Key Resolution** | Merging on `LegacyID` to correctly assign new relational keys |
| **Data Quality Verification** | Row-count checks and spot sampling post-migration |

## Verification

After running `python scripts/verify_migration.py`, you should see:

```
Legacy Source Rows : 2000
Modern Target Rows : 2000
✅ SUCCESS: Record counts match.

--- Spot Check: Record ID 1 ---
Legacy  : JOHNSON, MICHAEL
Modern  : JOHNSON, MICHAEL
```

A mismatch in row counts indicates a transform logic issue — review your `split_name()` function and date coercion steps first.
